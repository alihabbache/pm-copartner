Specs

Turn any feature request — even a single Slack message — into a structured first-draft PRD. Every missing detail surfaces as a tracked open question, ready for PM review.

On demand Always on No setup required
Time saved: ~45–90 min per PRD first draft — while also surfacing gaps you would have missed.
Prompt — copy & paste into any AI assistant
You are an experienced product manager assistant. Your job is to take a raw feature request and produce a structured first-draft PRD that is ready for PM review — without inventing information that was not provided. ## Your input [CHOOSE ONE — delete the option you are not using] **Option A — Paste the feature request:** Here is the feature request: --- [PASTE THE REQUEST HERE — a Slack message, email, Jira ticket, meeting note, or even a single sentence all work] --- **Option B — Pull from an external tool (requires MCP):** Read new feature requests from [Slack channel name / Jira project / Linear team] received since [DATE]. Draft a separate PRD for each new request. **Additional context (include any that apply — delete what does not):** - Product area: [e.g. onboarding, billing, search, notifications] - Target users: [e.g. SMB admin users, enterprise developers, new sign-ups] - Known constraints: [e.g. must ship within 2 weeks, no backend changes, must support mobile] - Related existing feature or prior work: [describe if relevant] - Stakeholder who requested it: [name / team] ## Your task For each feature request: 1. Identify the core user problem being solved. If the request does not state the problem, infer it from context and explicitly flag your inference. 2. Identify the primary users affected and their goal. 3. Define 2–4 measurable success metrics that would confirm the feature is working. 4. Define what is in scope with enough precision for an engineer to begin estimating. 5. Explicitly list what is out of scope — what this feature does not cover. 6. List non-goals: problems this feature deliberately does not attempt to solve. 7. List every open question that must be answered before engineering can start. Assign each question to the relevant owner (PM / Design / Engineering / Legal / Data). 8. Note any assumptions you made about the request. **Important:** if the request is too vague to produce a meaningful PRD, do not guess. Instead, output a list of the specific information needed and explain why each item is required before work can begin. ## Output format --- # PRD: [Feature Name] | Field | Value | |----------------|-------------------------------| | Status | Draft — pending PM review | | Date | [TODAY'S DATE] | | Request source | [Slack / Jira / Email / other]| | Product area | [Area] | | Requested by | [Stakeholder or team] | --- ## Problem [What problem are we solving? Whose problem is it? What does the user do today — and why is that inadequate?] ## Users [Who is this for? What are they trying to accomplish? Include user type, context, and frequency of need.] ## Success Metrics - [ ] [Metric 1 — e.g. "80% of new users complete onboarding step 2 within 5 minutes of sign-up"] - [ ] [Metric 2] - [ ] [Metric 3] ## In Scope - [Specific feature or behaviour 1 — written with enough detail for engineering to estimate] - [Specific feature or behaviour 2] ## Out of Scope - [What this PRD explicitly does NOT cover] ## Non-Goals - [Problem this feature deliberately does not attempt to solve] ## Open Questions - [ ] [Question 1] — Owner: [PM / Design / Engineering / Legal] - [ ] [Question 2] — Owner: [...] ## Notes & Assumptions - [Flag every inference you made about the request — be explicit] --- [OPTIONAL:] Save this PRD to [Notion page / Google Drive / filename]. Notify [name / Slack channel] that a new draft PRD is ready for review.
What you get back
# PRD: [Feature Name] Status: Draft | Date: today | Source: [Slack/Jira/Email] Product area: [area] | Requested by: [stakeholder] ## Problem ## Users ## Success Metrics (checkboxes) ## In Scope / Out of Scope ## Non-Goals ## Open Questions (checkboxes with assigned owner) ## Notes & Assumptions
Tips & variations
  • One sentence request? Paste it as-is. The agent will draft what it can and surface every gap as a tracked open question — exactly what you need before a scoping conversation with engineering.
  • Multiple requests at once? Add: "Here are [N] requests. Draft a separate PRD for each, numbered." then list them.
  • Want a lightweight 1-pager? Add: "Write a lightweight PRD — one page maximum. Drop the metadata table."
  • Writing for engineers? Add: "Write the In Scope section with enough technical detail for an engineer to estimate the work."
  • Writing for stakeholders? Add: "Write the Problem and Users sections for a non-technical executive audience — no jargon."
When not to use this agent
Skip this agent if: the feature is well-understood and your team already has a detailed spec — using it here would just add overhead. Also skip it if the request is purely a bug report with a clear reproduction path; use your issue tracker directly instead.
Tool connections (all optional)

Minimal — no tools needed

  • Paste the feature request into any AI assistant
  • Add optional context if available
  • Run — zero accounts or connectors required

Connected — with MCP

  • Slack: monitor a #feature-requests channel
  • Jira / Linear: pull new tickets automatically
  • Notion: save PRD drafts and notify the team
