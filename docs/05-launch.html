Launch

Turn your sprint output into polished, user-facing release notes — written in your brand voice, grouped by impact, with every item framed around what it means for the user.

Per sprint close Per release No setup required
Time saved: ~30–60 min of copywriting per release — plus no more engineer-written changelogs.
Prompt — copy & paste into Claude
You are a product communications specialist helping a Product Manager write clear, user-facing release notes that people will actually read. ## Your inputs **What shipped this sprint or release:** [CHOOSE ONE — delete the option you are not using] Option A — Paste your sprint output: --- [PASTE CLOSED TICKETS, A CHANGELOG, OR A BULLET LIST HERE] Prefix each item with its type: - [FEAT] Redesigned onboarding flow with a step-by-step progress indicator - [FEAT] Added CSV export to all Analytics report views - [FIX] Login button unresponsive on iOS 16 - [FIX] Incorrect date displayed in timezone conversion - [CHORE] Upgraded internal logging library — no user-facing impact --- Option B — Pull from a tool (requires MCP): Read all closed tickets from [Jira sprint name / Linear cycle / GitHub milestone] for this release. **Your brand voice:** [DESCRIBE YOUR TONE — or delete this block entirely and the agent will default to clear, friendly, and jargon-free] Examples: - "Professional but human. No buzzwords. Write as if you are talking to a smart non-technical user." - "Concise and direct. Our audience includes engineers and PMs. Technical terms are fine." - "Warm and encouraging. Our users are small business owners, not developers." Or: Read our tone-of-voice guidelines from [Notion page / Drive doc]. **Release details:** - Product name: [YOUR PRODUCT NAME] - Version or sprint label: [e.g. v2.4 / Sprint 18 / October 2025 Release] - Release date: [DATE] - Primary audience: [External customers / Internal team / Both] ## Your task 1. Read every item carefully. 2. Discard internal-only items with no user-facing impact — infrastructure changes, code refactors, dependency upgrades, and internal tooling. Do not mention these in the output. 3. For each user-facing item: - Write a clear, human title. Do not copy the ticket title — write it the way a user would describe the change. - Write 1–2 sentences: what changed, and specifically why the user will care or what problem it solves for them. - Match the brand voice provided above. 4. Group items: features and improvements together first, bug fixes second. 5. Write a single opening sentence summarising the whole release in plain language. 6. For any item where you are uncertain whether it has user-facing impact, include it but mark it [NEEDS PM REVIEW]. ## Output format --- # [PRODUCT NAME] Release Notes — [VERSION / SPRINT] **Released:** [DATE] ## What's New [One plain-language sentence summarising the release.] ### New Features & Improvements #### [User-facing title — not the ticket title] [1–2 sentences. What changed. Why the user will care. Written in the brand voice above.] #### [User-facing title] [...] ### Bug Fixes - **[Fix title]:** [One sentence — what was wrong and what is fixed.] - **[Fix title]:** [...] --- *Questions or feedback? [SUPPORT CHANNEL / EMAIL]* --- [OPTIONAL:] Save this to [Google Drive / Notion / filename]. Send via [Gmail / email] to [recipient list]. Post for internal review before publishing externally.
What you get back
# [Product] Release Notes — [Version / Sprint] Released: [Date] ## What's New (one-sentence plain-language summary) ### New Features & Improvements #### [User-facing title — written for users, not engineers] [1–2 sentences. What changed. Why the user cares. In brand voice.] ### Bug Fixes - **[Fix title]:** one sentence — what was wrong, now fixed. --- *Questions? [support channel]*
Tips & variations
  • No ticket system? List what shipped as bullet points from a meeting note or memory — the agent works with loose, unstructured input.
  • Mixed audience? Add: "Write two versions: one for external users (plain language, user-benefit focus) and one for the internal team (technical detail, includes chores and infra)."
  • Want it shorter? Add: "Limit to 300 words maximum. Prioritise the highest-impact changes."
  • Want social-ready copy? Add: "After the release notes, write a 2–3 sentence version suitable for a LinkedIn post or an in-app banner announcement."
  • Nothing user-facing this sprint? Add: "If there are no user-facing changes, write a short internal update — what was worked on and what is coming next sprint."
When not to use this agent
Skip this agent if: you have no closed tickets or completed work to describe — it can not invent what shipped. Also avoid using it for major version announcements that require legal or marketing review before release; use it as a first draft, not a publish-ready document.
Tool connections (all optional)

Minimal — no tools needed

  • Paste your closed tickets or sprint summary
  • Describe your brand voice in one sentence
  • Run — zero accounts or connectors required

Connected — with MCP

  • Notion: read tone-of-voice guide + save the notes
